## 3. PROTECTED FILES — НЕ ТРОГАТЬ БЕЗ ПИСЬМЕННОГО РАЗРЕШЕНИЯ

Эти куски кода — результат >50 итераций с владельцем. Любая «улучшающая» правка ломает то, что выстраивалось вручную.

### 3.1 Hero-мессенджеры на главной (`index.html` + `css/premium-overrides.css`)

Должны оставаться:
- Круглыми inline-SVG иконками
- С кольцевым названием вокруг (`textPath`)
- При hover — плоское название подлетает вверх, кольцевое уходит
- Цвета строго: WhatsApp `#25D366`, Telegram `#229ED9`, MAX `#7B5EE8` / `#8e74ee`

**Запрещено:**
- Превращать их в pill-кнопки с текстом
- Перекрашивать в белый или любой другой цвет
- Удалять классы: `messenger-group--ring`, `btn-hero-ring`, `.hero-ring-text`, `.hero-flat-text`
- Удалять CSS-блок `AI DO NOT TOUCH — R10 PROTECTED BLOCK` в `premium-overrides.css`
- Удалять CSS-блок `PATCH R9/R10`

### 3.2 Lightbox в пригородах (`prigorody/_template.html`)

```html
<div class="lightbox" id="lightbox">
```

Без него фото тортов в каталоге пригородов не открываются. **Не удалять.**

### 3.3 Иконка телефона в `#contacts`

`#contacts .contact-primary-icon svg` — чисто чёрная с мягкой gold-подсветкой. **Не возвращать белую/случайную заливку.**

### 3.4 Footer-контакты

Спокойные neutral premium chips. **Не раскрашивать в яркие цвета.**

### 3.5 Subtitle-типографика

`.hero-subtitle`, `.section-sub` — Cormorant Garamond normal без курсива. **Не заменять на грубый sans.**

### 3.6 Hero messenger hover JS-fallback (`js/main.js`)

Рядом с комментарием `R15: main page uses class-based SVG text` — поддержка class-based (главная) и id-based (пригороды) SVG. **Не удалять.**

### 3.7 buildCartKey / parseCartKey (`js/main.js`, r18)

Единый формат `id:mode:fill:decor` для всех товаров. **Не переписывать в "более простой" формат** — это ломает корзину для bento-режимов.

### 3.8 Автотесты protected UI

Protected UI теперь проверяется двумя слоями:

- статически: `scripts/audit.py` проверяет наличие критичных классов, функций и CSS-маркеров;
- в браузере: `tests/protected-interactions.spec.js` проверяет hover-анимацию WhatsApp / Telegram / MAX, отзывы и модалки.

Если меняешь protected-зону — `npm run qa` обязателен, а не опционален.

## 4. CSS-ПРАВИЛА — железобетонные

### 4.1 Каскад

Правильный порядок загрузки CSS в `<head>` (НЕ МЕНЯТЬ):

1. `style.css` — базовый
2. `mc-2026.css` — премиум-слой
3. `premium-overrides.css` — финальные правки + защищённые блоки
4. `v20-dark-and-fixes.css` — тёмная тема
5. `v20-fixes.css` — точечные хотфиксы
6. `final-fixes.css` — финальные hero/CTA правки

Каждый последующий перекрывает предыдущий по специфичности или порядку. **Не менять порядок** без анализа последствий.

### 4.2 `!important`

Сейчас в репо **1458 использований `!important`**. Это плохо. Новые правки:
- Сначала пытайся обойтись более специфичным селектором
- Если уже есть `!important` на том же селекторе — НЕ добавлять ещё один поверх, а **исправить существующий**
- Если без `!important` никак — оставить комментарий `/* important: причина */` рядом

### 4.3 Тёмная тема

Все цвета — через CSS-переменные в `:root` и `[data-theme="dark"]`:
- `--cream`, `--cream-dark`, `--gold`, `--brown`, `--text`, `--text-muted`, `--white`, `--bg-dark`, `--card-bg`

**Запрещено хардкодить цвета поверх токенов.** Если нужен новый оттенок — добавь токен в обе темы.

### 4.4 Где править — таблица решений

| Что правишь | В какой файл | Не в какой |
|---|---|---|
| Базовая верстка / цвета / шрифты | `style.css` | не делать дубль в `premium-overrides` |
| UX / анимация / accessibility | `mc-2026.css` | не плодить новые fix-файлы |
| Конкретный баг (1 селектор) | `v20-fixes.css` (если коротко) или `premium-overrides.css` | не создавать новый CSS |
| Тёмная тема | `v20-dark-and-fixes.css` | не дублировать в `style.css` |
| Hero / CTA полировка | `final-fixes.css` | не размазывать по другим |
| Галерея | `css/gallery/gallery-2026.css` | не лезть в основные CSS |

### 4.5 CSS/JS budget policy

- Runtime budget считается по gzip transfer size, а не raw bytes.
- Raw CSS/JS totals в `scripts/audit.py` — INFO/reference, не deploy-blocker.
- `!important` counts зафиксированы baseline-бюджетом в `scripts/audit.py`.
- Новые `!important` сверх baseline — warning и требуют осознанного объяснения.
- Массово удалять `!important` запрещено: сначала определить protected-зону, затем маленький патч, затем `npm run qa`.

## 6. ВЕРСИОНИРОВАНИЕ — синхронно или никак

### 6.1 Правило

Все ссылки на CSS/JS используют `?v=...`:
```html
<link rel="stylesheet" href="/css/style.css?v=20260517r19">
<script src="/js/main.js?v=20260517r19"></script>
```

И `sw.js`:
```js
const CACHE_NAME = 'milovi-cake-v2026.05.17-r19';
const PRECACHE = [
  '/css/style.css?v=20260517r19',
  '/js/main.js?v=20260517r19',
  // ...
];
```

### 6.2 После любой правки CSS или JS

Поднять версию **ВЕЗДЕ** одновременно. Найти все вхождения:

```bash
grep -rn '?v=' --include="*.html" --include="*.js"
```

И заменить на новую. Все три места:
- HTML-страницы (все, включая 14 пригородов)
- `sw.js` PRECACHE array
- `sw.js` CACHE_NAME

### 6.3 Запрещено

- ❌ Менять версию только в одном файле
- ❌ Использовать разные версии для разных CSS в одном HTML
- ❌ Забывать `sw.js`

## 11. ВИЗУАЛЬНЫЙ КОНТРОЛЬ — Playwright НЕ ДОСТАТОЧНО (КРИТИЧНО)

> **Главная боль владельца:** предыдущие агенты прогоняли десятки Playwright-тестов, отчитывались «61 passed», но в реальной мобильной версии оставались **очевидные баги, которые видно невооружённым глазом**: обрезанная кнопка «Добавить в корзину», цена наезжает на кнопку, полупрозрачная нижняя панель поверх текста, дубли элементов. **Так нельзя. Это проходит мимо носа, как верблюд.**

### 11.1 Почему Playwright их пропускал
Стандартные проверки типа `expect(locator).toBeVisible()` или сравнение скриншотов с baseline ловят только **регрессии относительно предыдущего состояния**. Они НЕ ловят:
- текст, который **не влезает** в кнопку и обрезается (`overflow` / `text-overflow`);
- элементы, которые **наезжают** друг на друга (overlap), если оба «видимы»;
- полупрозрачные слои, сквозь которые **просвечивает** контент;
- баг, который был в baseline **с самого начала** (сравнение с кривым baseline = «всё ок»);
- проблемы только на конкретной ширине (напр. 561–768px — зона 2-колоночного каталога).

### 11.2 ОБЯЗАТЕЛЬНЫЙ протокол визуальной проверки (каждая правка вёрстки)
1. **Снимай реальные скриншоты на нескольких ширинах**, а не только 375px:
   `360, 390, 414, 561, 600, 768, 900, 1024` (× light/dark). Зона **561–768px особенно коварна** — там каталог = 2 узкие колонки.
2. **Смотри скриншоты глазами**, описывай что видишь, а не только «тест зелёный». Спрашивай себя по каждому экрану:
   - Влезает ли весь текст в кнопки/бейджи? Нет ли `…` обрезки?
   - Не наезжают ли элементы друг на друга? (цена ↔ кнопка, бейдж ↔ иконка)
   - Нет ли просвечивания сквозь фиксированные панели (header, bottom-nav)?
   - Нет ли дублей (две кнопки «наверх», два бейджа)?
   - Ровные ли отступы, одинаковы ли радиусы и тени между карточками?
3. **Добавляй content-aware guard'ы** в `scripts/audit.py` / Playwright, которые проверяют СУТЬ, а не факт видимости:
   - `scrollWidth <= clientWidth` для кнопок (нет обрезки текста);
   - bounding-box'ы цены и кнопки **не пересекаются**;
   - `getComputedStyle(bottomNav).backgroundColor` имеет alpha ≥ 0.95 (панель непрозрачна);
   - в каталоге `≤768px` футер карточки `flex-direction: column` (нет бокового clipping).
4. **Не доверяй baseline вслепую.** Если baseline-скрин сам кривой — сначала почини реальность, потом пересними baseline. Никогда не «фиксируй» баг как эталон.

### 11.3 Правило «верблюда»
Если на скриншоте есть дефект, который **видно с первого взгляда обычному человеку** — он НЕ ДОЛЖЕН пройти ревью, даже если все автотесты зелёные. Зелёные тесты ≠ красивый сайт. **Финальный критерий — глаз, а не счётчик passed.**

### 11.4 Известные коварные зоны (проверять всегда)
| Зона | Что ломается | На что смотреть |
|---|---|---|
| Каталог 561–768px | 2 узкие колонки + горизонтальный футер → обрезка CTA | футер должен быть `column` ≤768px |
| `.mc-nav` (нижняя панель) | прозрачность → просвечивает контент | alpha фона ≥0.95 + `.has-mc-nav` padding на body |
| `.back-to-top` | наезд на нижнюю панель | `bottom: calc(82px + safe-area)` |
| Бейджи «ХИТ»/«ОРЕХИ», цена-цифра | наезд на текст/кнопку | bounding-box не пересекается |
| Длинные слова («Добавить в корзину», названия) | overflow | влезает или переносится, без `…` |
| iPhone notch / Android nav | контент под системными барами | `env(safe-area-inset-*)` |

## 12. ЧТО МОЖНО ТРОГАТЬ И В КАКУЮ СТОРОНУ

> Раздел [§3 PROTECTED] — это «не трогать». Здесь — наоборот: **где разрешено и поощряется улучшать**, и в каком направлении. Философия владельца: **тихий премиум-минимализм** — тёплое золото/коричневый, спокойные анимации, ровная «ювелирная» вёрстка. Улучшать ГЛУБИНУ вида, а не громкость эффектов.

### 12.1 Можно трогать (улучшать вид), сохраняя стиль
| Зона | Можно | Направление |
|---|---|---|
| Карточки каталога | отступы, радиусы, тени, ритм | ровнее, спокойнее, одинаковее. Без «парения» |
| Витрина товаров | новые фото, **видео** товара, лучший layout | живее и качественнее (см. §12.3) |
| Tap-feedback | единый `scale`, мягкая пружина | один стиль на всё, без «дёрганья» |
| Типографика | межбуквенный интервал, размеры, line-height | чище, читабельнее |
| Тёмная тема | контраст, ровность токенов | проверять обе темы всегда |
| Микро-переходы | длительность/easing (`--ease-premium`) | плавно, незаметно, дорого |

### 12.2 Нельзя вводить (это «не наш стиль», уже откатывали)
- Холодные/синие акценты КАК ЗАМЕНУ золота (⚠️ синий **wave-text hover** в hero/контактах — это ОРИГИНАЛ владельца с первого коммита, его НЕ удалять и НЕ считать багом; речь о новых чужих синих).
- Shimmer/glow/«блик пробегает», свечение цен, 3D-rotate карточек, magnetic hover на всё, parallax, frosted-glass blur на мобильную навигацию, кастомный золотой скроллбар, View Transitions API, prefetch-on-hover. Всё это — «luxury bloat», уже удаляли в r62–r66.
- Рост `!important`-долга сверх baseline-бюджета (audit это ловит).
- Новые CSS/JS файлы (структура зафиксирована, [§2]).

### 12.3 Премиум-фичи, которые впишутся красиво (идеи на будущее)
Реализовывать **только** в существующих файлах, по согласованию, с проверкой §11.

1. **Бегущая бесшовная видео-полоса** на главной (узкая лента «наши работы в движении»).
   - Постеры по умолчанию (лёгкие `.webp`), реальное `<video>` грузится/играет только у наведённого ролика.
   - «Магнитный» hover: наведённый ролик расширяется, соседи приглушаются. На тач — по тапу.
   - Данные — из готового `js/gallery/data.js` (16 видео уже в `/img/gallery/videos/`).
   - **Рабочий прототип:** `_mockups/video-strip-demo.html` (открыть в браузере). Разметка → `index.html`, стили → `mc-2026.css`, логика → IIFE в `mc-2026.js`.
2. **Витрина товара с видео** — в карточке каталога опционально `<video muted loop playsinline>` вместо/поверх фото (по тапу), постер — текущее фото. Тяжёлые файлы — lazy + только в видимой карточке.
3. **Lightbox с видео** в галерее (если ещё не везде) — плавное открытие, свайп между работами.
4. **Скелетоны загрузки** для медиа — мягкое золотистое мерцание (subtle, не «дискотека»).
5. **Sticky mini-CTA** на длинных лендингах — деликатная кнопка «Заказать», появляется после первого экрана (без перекрытия контента, с safe-area).

> Любая фича сначала — как прототип в `_mockups/` или на отдельной ветке, показать владельцу, и только потом интеграция. «Тихо и дорого» > «громко и дёшево».
