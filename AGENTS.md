## 3. PROTECTED FILES — НЕ ТРОГАТЬ БЕЗ ПИСЬМЕННОГО РАЗРЕШЕНИЯ

Эти куски кода — результат >50 итераций с владельцем. Любая «улучшающая» правка ломает то, что выстраивалось вручную.

### 3.1 Hero-мессенджеры на главной (`index.html` + `css/premium-overrides.css`)

Должны оставаться:
- Круглыми inline-SVG иконками
- С кольцевым названием вокруг (`textPath`)
- При hover — плоское название подлетает вверх, кольцевое уходит
- Цвета строго: WhatsApp `#25D366`, Telegram `#229ED9`, MAX `#7B5EE8` / `#8e74ee`

**Запрещено:**
- Превращать их в pill-кнопки с текстом
- Перекрашивать в белый или любой другой цвет
- Удалять классы: `messenger-group--ring`, `btn-hero-ring`, `.hero-ring-text`, `.hero-flat-text`
- Удалять CSS-блок `AI DO NOT TOUCH — R10 PROTECTED BLOCK` в `premium-overrides.css`
- Удалять CSS-блок `PATCH R9/R10`

### 3.2 Lightbox в пригородах (`prigorody/_template.html`)

```html
<div class="lightbox" id="lightbox">
```

Без него фото тортов в каталоге пригородов не открываются. **Не удалять.**

### 3.3 Иконка телефона в `#contacts`

`#contacts .contact-primary-icon svg` — чисто чёрная с мягкой gold-подсветкой. **Не возвращать белую/случайную заливку.**

### 3.4 Footer-контакты

Спокойные neutral premium chips. **Не раскрашивать в яркие цвета.**

### 3.5 Subtitle-типографика

`.hero-subtitle`, `.section-sub` — Cormorant Garamond normal без курсива. **Не заменять на грубый sans.**

### 3.6 Hero messenger hover JS-fallback (`js/main.js`)

Рядом с комментарием `R15: main page uses class-based SVG text` — поддержка class-based (главная) и id-based (пригороды) SVG. **Не удалять.**

### 3.7 buildCartKey / parseCartKey (`js/main.js`, r18)

Единый формат `id:mode:fill:decor` для всех товаров. **Не переписывать в "более простой" формат** — это ломает корзину для bento-режимов.

## 4. CSS-ПРАВИЛА — железобетонные

### 4.1 Каскад

Правильный порядок загрузки CSS в `<head>` (НЕ МЕНЯТЬ):

1. `style.css` — базовый
2. `mc-2026.css` — премиум-слой
3. `premium-overrides.css` — финальные правки + защищённые блоки
4. `v20-dark-and-fixes.css` — тёмная тема
5. `v20-fixes.css` — точечные хотфиксы
6. `final-fixes.css` — финальные hero/CTA правки

Каждый последующий перекрывает предыдущий по специфичности или порядку. **Не менять порядок** без анализа последствий.

### 4.2 `!important`

Сейчас в репо **1458 использований `!important`**. Это плохо. Новые правки:
- Сначала пытайся обойтись более специфичным селектором
- Если уже есть `!important` на том же селекторе — НЕ добавлять ещё один поверх, а **исправить существующий**
- Если без `!important` никак — оставить комментарий `/* important: причина */` рядом

### 4.3 Тёмная тема

Все цвета — через CSS-переменные в `:root` и `[data-theme="dark"]`:
- `--cream`, `--cream-dark`, `--gold`, `--brown`, `--text`, `--text-muted`, `--white`, `--bg-dark`, `--card-bg`

**Запрещено хардкодить цвета поверх токенов.** Если нужен новый оттенок — добавь токен в обе темы.

### 4.4 Где править — таблица решений

| Что правишь | В какой файл | Не в какой |
|---|---|---|
| Базовая верстка / цвета / шрифты | `style.css` | не делать дубль в `premium-overrides` |
| UX / анимация / accessibility | `mc-2026.css` | не плодить новые fix-файлы |
| Конкретный баг (1 селектор) | `v20-fixes.css` (если коротко) или `premium-overrides.css` | не создавать новый CSS |
| Тёмная тема | `v20-dark-and-fixes.css` | не дублировать в `style.css` |
| Hero / CTA полировка | `final-fixes.css` | не размазывать по другим |
| Галерея | `css/gallery/gallery-2026.css` | не лезть в основные CSS |

## 5. JS-ПРАВИЛА

### 5.1 Архитектура

- Всё внутри `(function () { 'use strict'; ... })();` IIFE
- Глобал доступ через `window.functionName = functionName` в конце файла
- Inline `onclick="fn(...)"` в HTML работает только через `window.*`

### 5.2 Запреты

- ❌ Не создавать дубликаты функций (после правки — `grep "^function NAME" file.js | wc -l` должно быть `1`)
- ❌ Не «сжимать» функции в однострочники (это сломалось в b0b5ebe — см. историю)
- ❌ Не удалять `window.*` экспорты — это сломает onclick в HTML
- ❌ Не переименовывать функции без замены **во всех** HTML

### 5.3 Перед коммитом JS обязательно:

```bash
node --check js/main.js
node --check js/nav.js
node --check js/mc-2026.js
node --check js/v20-faq-fix.js
node --check js/gallery/main.js
node --check sw.js
```

Если хоть один FAIL — **не коммитить**.

## 6. ВЕРСИОНИРОВАНИЕ — синхронно или никак

### 6.1 Правило

Все ссылки на CSS/JS используют `?v=...`:
```html
<link rel="stylesheet" href="/css/style.css?v=20260517r19">
<script src="/js/main.js?v=20260517r19"></script>
```

И `sw.js`:
```js
const CACHE_NAME = 'milovi-cake-v2026.05.17-r19';
const PRECACHE = [
  '/css/style.css?v=20260517r19',
  '/js/main.js?v=20260517r19',
  // ...
];
```

### 6.2 После любой правки CSS или JS

Поднять версию **ВЕЗДЕ** одновременно. Найти все вхождения:

```bash
grep -rn '?v=' --include="*.html" --include="*.js"
```

И заменить на новую. Все три места:
- HTML-страницы (все, включая 14 пригородов)
- `sw.js` PRECACHE array
- `sw.js` CACHE_NAME

### 6.3 Запрещено

- ❌ Менять версию только в одном файле
- ❌ Использовать разные версии для разных CSS в одном HTML
- ❌ Забывать `sw.js`

## 7. ОБЯЗАТЕЛЬНЫЕ ПРОВЕРКИ перед коммитом

```bash
# 1. Синтаксис JS
node --check js/main.js
node --check js/nav.js
node --check js/mc-2026.js
node --check js/v20-faq-fix.js
node --check js/gallery/main.js
node --check sw.js

# 2. Генератор пригородов (если правил _template.html или _cities.csv)
python3 -m py_compile prigorody/build.py
python3 prigorody/build.py

# 3. Версии синхронизированы (все должны выдать одну и ту же строку)
grep -rohE '\?v=[0-9a-zA-Z]+' --include="*.html" --include="*.js" | sort -u

# 4. Нет дубликатов функций в JS
awk '/^function [a-zA-Z_]/{name=$2; sub(/\(.*/,"",name); print name}' js/main.js | sort | uniq -d
# (должно быть пусто)

# 5. CACHE_NAME в sw.js совпадает с ?v=
grep -E "CACHE_NAME|\?v=" sw.js
```

Если хоть одна проверка не прошла — **откатить и переделать**, не коммитить.

### 8.2 При правке

1. Минимально-инвазивно. Не «улучшать», что не просили.
2. Сохранять рабочий стиль кода файла.
3. Не «рефакторить» (переименовывать, перестраивать) без явного запроса.

### 8.3 После правки

2. Поднять `?v=` синхронно
3. В сообщении коммита указать: что и зачем меняли + какие проверки прошли

### 8.4 При откате чужой правки

1. НЕ удалять все правки оптом
2. Понять, какая конкретно правка добавлена, какая нужна, какая лишняя
